Verifiers & LLM-as-a-Judge

How do we score a reasoning model's answer? This page walks through both scoring paradigms from Build a Reasoning Model (From Scratch) — rule-based verifiers and the LLM-as-a-judge — with every mechanism wired to a live demo you can poke at. Everything here mirrors the runnable Python in llm_judge_and_verifiers.py.

01 · Orientation

Two ways to turn a generation into a number

A reasoning model emits free text. To evaluate it we need a scalar — correct/incorrect, a 1–5 quality score, or a preference. There are two families of scorer, and most of this seminar is about knowing which one a task calls for and what each one quietly gets wrong.

VERIFIER

Rule-based / programmatic. Extract the final answer, normalize, compare to ground truth. Output is a hard 0/1.

✓ cheap, deterministic, reproducible, no model bias
✗ needs a checkable answer · brittle to formatting · silent on reasoning quality & partial credit

LLM JUDGE

Model-based. A (stronger) LLM reads instruction + reference + candidate and emits a score or a preference.

✓ handles open-ended output, partial credit, style, helpfulness
✗ stochastic · biased (position/verbosity/self) · costs compute · must be validated vs humans

"…so x = \boxed{8}."
→
Scorer
verifier or judge
The rule of thumb for the whole talk: use a verifier whenever the task admits one (math, multiple choice, code with tests). Reach for a judge only when "correct" isn't mechanically checkable — and even then, anchor it with a reference answer and an explicit rubric. The chF/04 script does exactly this: a verifier defines "right," and the judge grades quality against a reference.

02 · Rule-based verifiers

Everything hides behind ==

A verifier is three steps: extract the final answer from the generation, normalize it, then compare to gold. The book ships two flavours: letter-matching for MMLU (grab the first A/B/C/D) and boxed/numeric extraction for MATH-500. Both look trivial — and both produce false negatives the moment formatting drifts. Try to break the grader below.

Verifier playground · extract → normalize → grade
Press “Grade it” to trace each step.
Why this matters for ML researchers: verifier false-negative rate is a confound in every "accuracy" number you read. A 2–3 pt gap between papers is often a normalization detail (units, $, \frac vs decimal, trailing period), not capability. When you report verifier accuracy, report the extraction rule too.

The extraction + normalization code (JS mirrors the Python) 
function extractBoxedOrNumber(text){
  // 1) prefer contents of the LAST \boxed{...}
  const boxed = [...text.matchAll(/\\boxed\{([^{}]*)\}/g)].map(m=>m[1]);
  if(boxed.length) return normalize(boxed.at(-1));
  // 2) else fall back to the last number in the text
  const nums = text.match(/-?\d[\d,]*\.?\d*/g);
  return nums ? normalize(nums.at(-1)) : null;
}
function normalize(s){            // strip $ , \  and trailing '.', fold 1000.0->1000
  s = s.trim().replace(/\$|,|\\/g,"").replace(/\.+$/,"");
  return String(Number(s) || s.toLowerCase());
}

03 · Self-consistency

Turn a verifier + sampling into accuracy

A verifier needs ground truth to score — but you can use it without ground truth to pick an answer. Sample the model N times at temperature > 0, extract each final answer, and take the majority vote (ch04 self_consistency_math500.py). This is test-time compute converting directly into accuracy: independent reasoning paths tend to agree on the right answer and scatter on wrong ones.

Self-consistency vote · majority over extracted answers
Each line is one model sample; we extract its boxed answer and vote.

Intuition for the seminar: if each independent sample is right with probability p > 0.5, majority vote over N samples concentrates toward correct as N grows (a Condorcet-jury effect). The win comes for free from a verifier you already have — no judge, no extra labels.

04 · The judge loop

What ollama-judge.py actually does

When answers are open-ended, swap the verifier for a judge model. The chF/04 script pits a local Qwen3 (base vs reasoning) against MATH-500, then has gpt-oss:20b grade each answer 1–5 against a rubric and a reference. Strip away the infrastructure and the loop is five steps:

#stepcode
1Build candidate answer from the model under testgenerate_with_qwen3()
2Assemble rubric prompt: instruction + reference + answerrubric_prompt()
3Send to judge (temp 0, fixed seed) via Ollamaquery_model()
4Parse a 1–5 integer out of the judge's textparse_score()
5Aggregate: average score + count distributionscore_counts
Design decisions baked into that loop — each is a knob worth discussing: the judge runs at temperature 0 + fixed seed for reproducibility; the prompt supplies a reference answer to anchor grading; the rubric is 1–5, not 1–10 (lower variance); and the candidate and judge are different model families to blunt self-preference bias.

The book's own run shows the headline result: the reasoning variant beats the base model under the same judge — average 4.30 vs 3.80 over 10 MATH-500 items. The judge is the measuring instrument; the next sections are about trusting it.

05 · Prompt construction

The rubric prompt, assembled live

The judge is only as good as its prompt. Edit the three inputs and watch the exact string that gets sent to the model assemble below — verbatim structure from rubric_prompt(). The mock score is a stand-in heuristic (lexical overlap with the reference → 1–5) so you can iterate without a GPU; a real judge reasons over the rubric instead.

rubric_prompt() builder
Why a reference answer + bounded scale. Without a reference, the judge silently re-derives "correct" and grades against its own (possibly wrong) belief — disastrous on hard math. A bounded 1–5 anchored rubric makes scores comparable across items and across runs, which is what lets you average them.

06 · Output parsing

Getting a number back out is its own failure mode

The judge returns prose. parse_score() grabs the first digit 1–5 followed by a non-digit: re.search(r"([1-5])(?:\D|$)", text). Simple — and exactly where silent corruption creeps in. If the judge restates the rubric ("on a scale of 1 to 5…"), writes "10/10", or leads with the year, the regex latches onto the wrong token. Paste judge text and watch what the parser extracts.

parse_score() inspector
Robust alternatives to discuss: constrain the judge to reply in a fixed schema (JSON {"score": n}), use a grammar/structured-decoding constraint, or read the model's logprobs over the tokens "1".."5" instead of sampling free text. The book keeps the regex for simplicity — know its blast radius.

07 · Failure modes

A judge is a biased instrument

Because the judge is itself an LLM, it carries systematic biases. Three are robust enough to measure in a seminar demo. The pairwise sandbox below uses a deliberately verbosity-biased mock judge (it prefers the longer answer) so you can see position bias by swapping the slot order.

Position bias

The same content scores differently as "A" vs "B". Detect by running both orderings and checking for flips.

Verbosity bias

Longer, more confident answers get higher scores independent of correctness. Length is a confound.

Self-preference / familiarity bias

A judge rates outputs from its own model family higher — which is why chF/04 uses a different family (gpt-oss) to judge Qwen3.

Position-bias probe · same two answers, both orderings
Mitigations (good discussion prompt): randomize or average over orderings; calibrate for length; use multiple judges and aggregate; few-shot the judge with anchored examples; and always report a human-agreement number alongside any judge-derived metric.

08 · Pairwise & leaderboards

From A-vs-B verdicts to a ranking

Pairwise judging ("which answer is better?") is more reliable than absolute scoring and is what powers Chatbot-Arena-style leaderboards (chF/03). Collect many pairwise verdicts, then fit a latent strength per model. The book shows two fits: Elo (online update) and Bradley-Terry (maximum-likelihood). Below, enter a win matrix and compute Bradley-Terry strengths.

Bradley-Terry from a win matrix

Cell (row, col) = times row model beat col model. Edit the wins; diagonal ignored.

Bradley-Terry models P(i beats j) = πᵢ/(πᵢ+πⱼ) and fits the strengths π by MLE (iterative). Elo is essentially its online stochastic-gradient cousin. Either way, the input is pairwise judge verdicts — so leaderboard quality is bounded by judge quality, which brings us to validation.

09 · Validation

An unvalidated judge is a vibe, not a metric

Before trusting judge scores, measure how well they agree with ground truth (a verifier, where one exists) or with human labels. Two standard checks: accuracy of the judge's pass/fail vs gold, and Cohen's κ for chance-corrected agreement. Toggle each item's gold label and the judge's verdict to see κ respond.

Reading κ like a referee: κ ≈ 1 perfect, ≈ 0 no better than chance, < 0 worse than chance. For LLM-judge work, people usually want κ vs humans in the "substantial" band (≈ 0.6–0.8) before reporting judge scores as a proxy for human preference. Always publish the κ next to the score.

10 · Decision guide

Which scorer for which task?
TaskDefault scorerWhy
Multiple choice (MMLU)verifier letter-matchExact gold exists; judge would only add noise & cost.
Math with final answer (MATH-500)verifier + self-consistencyCheckable; sampling+vote buys accuracy free.
Code generationverifier unit testsExecution is ground truth; judges miss subtle bugs.
Open-ended QA / explanationsjudge rubric + referenceNo mechanical "correct"; need partial credit & quality.
Ranking many modelsjudge pairwise → Elo/BTPairwise is reliable; fits a leaderboard.
RL reward signalverifier if possible, else judgeVerifiers are gameable-resistant; judge rewards can be hacked.
Hybrid is the norm in practice. chF/04 uses both at once: a verifier-style reference defines the right answer, and the judge grades quality against it. Use the verifier for the hard correctness gate; use the judge for the graded, open-ended layer on top — and validate the judge against the verifier wherever they overlap.

11 · Hands-on

Run it yourself

Every demo on this page mirrors llm_judge_and_verifiers.py (saved alongside this file). It runs with zero dependencies in mock mode, or against a real judge via Ollama — same wire format as the book's ollama-judge.py.

# 1) no dependencies — deterministic mock judge, full pipeline
python llm_judge_and_verifiers.py

# 2) against a real judge (needs: ollama serve + a pulled model)
pip install requests
ollama pull gpt-oss:20b
python llm_judge_and_verifiers.py --backend ollama --judge_model gpt-oss:20b

# 3) the book's original end-to-end script (needs Qwen3 ckpt + torch)
uv run ollama-judge.py --which_model reasoning --dataset_size 10

The script prints a per-example table (verifier vs judge vs gold), the self-consistency tally, agreement metrics (accuracy + Cohen's κ), and a position-bias flip rate — the same quantities you just explored interactively. Each PART in the source maps to a section above.
